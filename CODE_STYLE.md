# StructureClaw 代码风格

## 1. 文档目的

这份文档记录当前仓库的真实代码风格现状，以及在没有统一格式化配置前，协作者应遵循的约定。

当前仓库并不是完全统一风格的状态，因此不要为了“看起来一致”去大规模重排无关文件。

## 2. 通用原则

- 修改哪个文件，就优先遵循该文件现有风格。
- 优先做局部、可控改动，避免顺手触发大面积格式噪音。
- 命名应尽量体现结构工程领域语义，而不是泛化技术词。
- 重构时优先保护 API 契约、错误码和回归脚本行为。
- 只有在逻辑不够直观时才补充简短注释。

## 3. TypeScript 约定

### 通用

- 默认以 TypeScript `strict` 模式思维编写代码。
- 优先使用清晰的领域类型，少用 `any`。
- 必须使用 `any` 时，应限制在局部且有明确理由。
- 子项目已支持 `@/*` 别名时，可继续沿用。
- 函数职责保持聚焦，返回结构尽量稳定。

### Backend 风格

`backend/` 中的实际风格特征：

- 本地 ESM 导入通常带 `.js` 后缀
- 大多数文件使用分号
- 大多数场景使用单引号
- 路由层与服务层职责分离较明显

后端协作约定：

- 路由层负责协议和校验，编排逻辑尽量下沉到 service。
- 面向 API 的错误优先保留结构化错误码。
- 不要把复杂编排逻辑堆进单个 route handler。
- 遵循 [backend/.eslintrc.cjs](/home/openclaw/tmp/structureclaw/backend/.eslintrc.cjs) 中已有规则。

### Frontend 风格

`frontend/` 中的实际风格特征：

- 大多数源码文件不写分号
- 单引号占主流
- 以函数式 React 组件为主
- Tailwind 工具类与基础 UI 组件组合使用

前端协作约定：

- 展示逻辑优先放组件，复杂状态逻辑优先放 hooks 或 store slice。
- 优先复用 `src/components/ui` 下已有基础组件。
- 保持可访问性和可测试性，不要为了快速实现破坏语义结构。
- 遵循 [frontend/.eslintrc.json](/home/openclaw/tmp/structureclaw/frontend/.eslintrc.json) 中已有基线规则。

## 4. Python 约定

`core/` 中的实际风格特征：

- 类型标注有使用，但还未完全统一
- Pydantic 模型是核心数据边界
- 模块/接口层常有说明性 docstring
- 目前没有集中式格式化配置文件

Python 协作约定：

- 基本遵循 PEP 8，函数保持短小、可测。
- 在数据边界处优先使用明确模型，而不是松散嵌套字典。
- API 错误输出保持结构化、可预期。
- 如需整理格式，应兼容 [core/requirements.txt](/home/openclaw/tmp/structureclaw/core/requirements.txt) 中已存在的 `black` 依赖，但不要顺带重排无关模块。

## 5. 测试与验证

- Backend 变更：优先跑对应 Jest 测试和相关契约脚本。
- Frontend 变更：优先跑对应 Vitest 测试和 `type-check`。
- Core 变更：优先跑覆盖目标能力的回归或契约脚本。
- 先做针对性验证，再做跨模块回归，避免只凭肉眼判断通过。

## 6. 文件与命名

- 文件名应尽量反映领域职责。
- API 模块按资源或工作流组织，不要随意打散。
- 测试文件应遵循各子项目现有组织方式。
- Backend 与 Core 之间的错误码、字段命名应尽量保持一致。

## 7. 不建议的做法

- 未单独立项就做仓库级格式化
- 在同一子项目里混用另一子项目的格式习惯
- 随意改动响应 envelope
- 用模糊工具名掩盖真实工程语义

## 8. 后续标准化建议

建议后续单独推进一次风格治理任务：

- 明确前后端是否要统一到同一套 TypeScript 格式化方案
- 为 Python 增加集中式工具配置
- 在统一 formatter 落地前，继续以“尊重局部现状”为主
