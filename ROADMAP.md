# StructureClaw 路线图

## 1. 当前阶段

截至 2026 年 3 月 11 日，StructureClaw 已经具备可运行的 MVP 基础：

- 本地与 Docker 启动路径已经打通
- Agent 编排入口已经存在
- 结构校验、格式转换、结构分析、报告输出已经串联
- 仓库内已有回归脚本与接口契约验证脚本

因此后续重点不应再放在“重新搭骨架”，而应放在标准化、稳定性和产品深度建设上。

## 2. 近期优先级

### P0. 工程标准化

目标：降低仓库协作成本，让贡献方式更可预测。

交付项：

- 根目录入门文档与项目摘要
- 明确的代码风格与技术栈说明
- 常用开发命令的统一入口说明
- 每个子项目更清晰的 lint、type-check、test 预期

完成标准：

- 新成员可只靠根目录文档理解项目结构
- 代码评审标准有据可依
- 风格要求不再依赖口头约定

### P1. Agent 工作流可靠性

目标：让编排结果更稳定、更容易排查。

交付项：

- 更稳定的工具调用错误码体系
- 更清晰的缺参澄清状态管理
- 更完整的流式与执行模式回归覆盖
- 更一致的报告与工件生成行为

完成标准：

- 常见失败场景可映射到稳定错误码
- 编排结果始终携带 trace 和 metrics
- 缺参澄清流程有脚本或测试覆盖

### P2. 结构模型治理

目标：让 `StructureModel v1` 真正成为单一事实来源。

交付项：

- Schema 演进与版本治理规则
- `v1.x` 迁移兼容策略说明
- 转换器契约基线与覆盖目标

完成标准：

- Schema 变更必须附带版本说明
- 迁移信息具备可追溯性
- 转换器 round-trip 质量可持续验证

## 3. 中期优先级

### P3. 分析能力扩展

目标：把当前 MVP 的分析能力向更实用的工程范围推进。

交付项：

- 更完整的 3D 分析支持
- 更丰富的荷载、组合和边界条件处理
- 更强的包络结果与控制工况输出
- 更多黄金算例回归

### P4. 规范校核与报告质量

目标：让输出更适合工程复核和交付。

交付项：

- 扩展校核规则覆盖面
- 提升“结果 -> 条文/输入/公式”的可追溯性
- 完善 JSON/Markdown 报告模板

### P5. 前端产品化

目标：从“调试控制台”向“可操作产品界面”过渡。

交付项：

- 更好的执行链路、工件、错误诊断 UI
- 更清晰的项目、模型、任务导航
- 更适合阅读的分析与报告展示

## 4. 长期优先级

### P6. 生产可用性

- 为主要回归套件建立 CI 门禁
- 建立分环境部署方案
- 明确观测、重试、超时、后台任务策略
- 建立发布与迁移治理流程

### P7. 生态与互操作

- 增加更多结构文件格式转换器
- 预留插件化扩展点
- 加深与外部工程软件链路的互操作能力

## 5. 建议执行顺序

1. 先补齐文档与协作规范
2. 再强化 Agent 编排契约与稳定性
3. 然后固化 Schema 与转换治理
4. 再扩展分析能力与回归深度
5. 然后提升校核与报告质量
6. 最后推进前端产品化与部署能力

## 6. 建议观察指标

- 新成员理解仓库所需时间
- 评审中风格类问题占比
- Backend/Core/Frontend 回归通过率稳定性
- Agent 编排结果可预测性
- 从用户请求到工程工件的可追溯程度
